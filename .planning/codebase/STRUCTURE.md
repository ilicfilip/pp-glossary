## Directory Purposes

**`includes/`:**
- Purpose: All plugin logic organized as classes by responsibility
- Contains: PHP class files following class-per-file pattern
- Key files: `class-post-type.php`, `class-meta-boxes.php`, `class-content-filter.php`, `class-settings.php`
- Naming: `class-{name}.php` maps to `PP_Glossary\{Name}` via autoloader
- Initialization: Each class has static `init()` method called from main plugin file

**`blocks/glossary-list/`:**
- Purpose: Gutenberg block registration and editor interface
- Contains: Block metadata (JSON) and editor script (vanilla JS)
- Key files: `block.json` (declares block, supports), `editor.js` (editor UI only)
- No frontend rendering here - server-side rendering happens in `class-blocks.php`

**`assets/css/`:**
- Purpose: All stylesheet definitions for popover appearance and glossary layout
- Contains: CSS Anchor Positioning rules for automatic positioning, CSS custom properties
- Key file: `glossary.css`
- Scope: Applies to `<dfn>`, `<button>`, `<aside popover>` elements, glossary block styling

**`assets/js/`:**
- Purpose: Frontend JavaScript for interactive behavior
- Contains: Click handlers, keyboard navigation (Enter/Space/Escape), aria-expanded toggling
- Key file: `glossary.js`
- Scope: Attaches to `[data-glossary-popover]` buttons, manages popover API calls
- Note: Deferred loading via Assets class - only loads if terms found in content

**`.planning/codebase/`:**
- Purpose: Architecture analysis documents generated by GSD mapping
- Contains: ARCHITECTURE.md, STRUCTURE.md (this file), and focus-specific docs
- Generated: Not checked in to repository; created by mapping agent
- Used by: Future GSD planning and execution phases

## Naming Conventions

**Files:**
- `class-{name}.php` - Class definition file (lowercase, hyphens)
  - Example: `class-content-filter.php` contains `PP_Glossary\Content_Filter`
- `block.json` - Block metadata (WordPress standard)
- `editor.js` - Block editor JavaScript (WordPress standard)

**Directories:**
- `includes/` - Core plugin classes
- `blocks/{name}/` - Individual block directories by block name (WordPress standard)
- `assets/{type}/` - Assets grouped by type (css, js)

**Classes:**
- `PP_Glossary\{Name}` - Namespace-based with underscores for multi-word names
  - Example: `PP_Glossary\Content_Filter`, `PP_Glossary\Post_Type`
- Static methods: `init()` for initialization, others as needed
  - Example: `Post_Type::register_post_type()`, `Settings::get_glossary_page_id()`

**Functions:**
- `pp_glossary_{action}()` - Global functions prefixed with plugin name + action
  - Example: `pp_glossary_init()`, `pp_glossary_autoloader()`, `pp_glossary_strtolower()`
- PascalCase in utility functions
  - Example: `pp_glossary_strtolower()`, `pp_glossary_substr()`

**Post Meta Keys:**
- `_pp_glossary_data` - Single consolidated array storing all entry fields
  - Used by: Meta_Boxes, Content_Filter, Blocks, Schema

**Post Type:**
- `pp_glossary` - Custom post type slug (lowercase with underscores)

**Block Names:**
- `pp-glossary/glossary-list` - Block namespace/name (WordPress standard format)

## Where to Add New Code

**New Feature (term behavior, filtering):**
- Primary code: `includes/class-content-filter.php`
- Tests: Create in `.planning/tests/` or alongside source
- Example: Adding synonym grouping or cross-references

**New Admin Setting:**
- Implementation: `includes/class-settings.php`
- Add to: `register_settings()` and `register_setting()` calls
- Getter method: Add static getter like `get_glossary_page_id()`
- Used by: Any class accessing settings via static getters

**New Meta Field (entry data):**
- Definition: Update `Meta_Boxes::get_entry_data()` defaults array
- UI: Update `Meta_Boxes::render_meta_box()` HTML
- Save: Update `Meta_Boxes::save_meta_boxes()` sanitization
- Migration: Add new `migrate_to_X_X_X()` if upgrading existing installations

**New Block Type:**
- Structure: `blocks/{block-name}/`
- Files: `block.json`, `editor.js`, render callback in `class-blocks.php`
- Registration: Call `register_block_type()` in `Blocks::register_blocks()`
- Render: Add method like `render_{block_name}_block()` returning HTML

**New Styling:**
- Location: `assets/css/glossary.css`
- Scope: Target via class names (e.g., `.pp-glossary-term`, `.pp-glossary-block`)
- Variables: Use CSS custom properties for theming (e.g., `--glossary-bg-color`)

**New JavaScript Interaction:**
- Location: `assets/js/glossary.js`
- Scope: Attach to elements via data attributes or class selectors
- Enqueueing: Update `Assets::enqueue_assets()` if new script needed

**Utility Function:**
- Location: `includes/functions.php`
- Pattern: Prefix with `pp_glossary_`
- Example: `pp_glossary_strtolower()`, `pp_glossary_substr()`

## File Inclusion Pattern

**Autoloading:**
- Main file loads functions: `require_once PP_GLOSSARY_PLUGIN_DIR . 'includes/functions.php'`
- Classes loaded via autoloader on demand: `spl_autoload_register( 'pp_glossary_autoloader' )`
- Map: `PP_Glossary\{Name}` → `includes/class-{name}.php` (converted to lowercase with hyphens)

**Block Registration:**
- Block JSON file loaded by WordPress via `register_block_type()` call with directory path
- Editor JS registered as `pp-glossary-block-editor` handle, enqueued with block type

**Assets Enqueueing:**
- CSS/JS only loaded if `Content_Filter::$terms_found_on_page` is true
- Prevents unnecessary loading when no glossary terms exist in content
